🚀 GitHub 팀 마스터 가이드

5명 팀을 위한 완벽한 GitHub 협업 가이드

1
Git 설치 및 초기 설정 (Windows)

🔧 Git 설치하기

  1. Git 다운로드: git-scm.com 접속
  2. Download for Windows 클릭
  3. 설치 시 모든 옵션 기본값으로 진행
  4. 설치 완료 후 Git Bash 실행
# 설치 확인 git --version
VS Code 터미널에서 Git 명령어가 안 된다면?
1. VS Code 완전 재시작
2. 터미널 우측 상단 드롭다운에서 "Git Bash" 선택
3. Ctrl + , → 검색: git.path → 값: C:\Program Files\Git\bin\git.exe

👤 Git 사용자 설정 (필수!)

# 사용자 정보 설정 git config --global user.name "당신의 이름" git config --global user.email "이메일@example.com" # 기본 브랜치 이름 설정 git config --global init.defaultBranch main # 설정 확인 git config --list
GitHub 계정과 동일한 이름/이메일로 설정하면 커밋이 제대로 연결됩니다!

✅ 2단계 체크리스트

3
로컬 저장소 연결하기

💻 방법 1: 새 프로젝트 시작하기

# 프로젝트 폴더 생성 mkdir team-project cd team-project # Git 저장소 초기화 git init # README.md 파일 생성 echo "# Team Project" > README.md # 스테이징 및 커밋 git add . git commit -m "첫 커밋" # 원격 저장소 연결 (HTTPS 방식) git remote add origin https://github.com/username/team-project.git # 푸시 git push -u origin main

💻 방법 2: 기존 저장소 클론하기 (추천)

# 팀 저장소 클론 git clone https://github.com/username/team-project.git cd team-project # 파일 수정 후 바로 작업 가능 echo "팀원 추가 내용" >> README.md git add . git commit -m "팀원 작업 추가" git push origin main

🔧 자주 발생하는 오류들

error: src refspec main does not match any
→ 해결: 첫 커밋을 하지 않았거나 브랜치 이름이 다름
→ git branch로 현재 브랜치 확인 후 git push -u origin 브랜치명
Updates were rejected because the remote contains work
→ 해결: git pull origin main --allow-unrelated-histories
→ 또는 git push origin main --force (주의!)
Permission denied (publickey)
→ 해결: HTTPS 방식 사용 (git remote set-url origin https://github.com/...)
팀 프로젝트에서는 Clone 방식을 추천합니다!
더 간단하고 충돌이 적어요.

4
기본 Git 워크플로우 마스터

🔄 기본 명령어 5개 (반드시 암기!)

git add . # 모든 변경사항 스테이징 git commit -m "메시지" # 커밋 (로컬에 저장) git push origin main # 원격 저장소에 업로드 git pull origin main # 원격 저장소에서 최신 코드 받기 git status # 현재 상태 확인

📋 일반적인 작업 흐름

1
작업 시작 전: git pull origin main
2
파일 수정/추가
3
git add . (스테이징)
4
git commit -m "의미있는 메시지"
5
git push origin main

💬 좋은 커밋 메시지 작성법

# 좋은 예시 git commit -m "로그인 기능 추가" git commit -m "헤더 스타일 수정" git commit -m "사용자 인증 버그 수정" # 나쁜 예시 git commit -m "수정" git commit -m "버그픽스" git commit -m "작업완료"
팀 협업 시 중요한 점:
• 매일 작업 시작할 때 git pull 먼저!
• 자주 커밋하고 푸시하기
• 의미있는 커밋 메시지 작성

✅ 4단계 체크리스트

5
브랜치로 안전하게 협업하기

🌿 브랜치가 필요한 이유

❌ 브랜치 없이 작업하면?

모든 팀원이 main에서 직접 작업 → 충돌 지옥 → 사이트 망가짐

✅ 브랜치로 작업하면?

각자 별도 브랜치에서 안전하게 작업 → 완성 후 병합 → 안정적인 협업

🔧 브랜치 기본 명령어

# 새 브랜치 생성 및 이동 git checkout -b feature/login # 브랜치 목록 확인 git branch # 다른 브랜치로 이동 git checkout main # 브랜치 병합 git merge feature/login # 브랜치 삭제 git branch -d feature/login

👥 팀 브랜치 전략

main # 🏠 항상 안정된 배포 가능한 코드 ├── feature/login # 👤 철수: 로그인 기능 ├── feature/payment # 💰 영희: 결제 기능 ├── feature/ui-design # 🎨 민수: UI 디자인 ├── bugfix/header # 🐛 지영: 헤더 버그 수정 └── feature/database # 🗃️ 동현: 데이터베이스 연동

🔄 브랜치 작업 워크플로우

# 1. 최신 main 코드 받기 git checkout main git pull origin main # 2. 새 브랜치 생성해서 작업 시작 git checkout -b feature/my-feature # 3. 작업 후 커밋 git add . git commit -m "새 기능 추가" # 4. 원격 저장소에 브랜치 푸시 git push origin feature/my-feature # 5. GitHub에서 Pull Request 생성 # 6. 코드 리뷰 받고 승인되면 병합
브랜치 작업 시 주의사항:
• 브랜치 이름은 명확하게 (feature/login, bugfix/header)
• 작업 완료 후 반드시 main으로 돌아가기
• 병합 완료된 브랜치는 삭제하기

✅ 5단계 체크리스트

6
Pull Request와 코드 리뷰

🔄 Pull Request란?

"내가 만든 기능을 main 브랜치에 합쳐달라"고 요청하는 것

✅ 코드 리뷰 가능
✅ 변경사항 추적 가능
✅ 토론 및 피드백 가능
✅ 안전한 병합 과정

📋 Pull Request 생성 과정

# 1. 브랜치에서 작업 완료 후 푸시 git push origin feature/my-feature # 2. GitHub 웹사이트에서 확인 # → 노란색 "Compare & pull request" 버튼 클릭 # → 또는 Pull requests 탭 → New pull request

📝 Pull Request 작성 팁

1
제목: 명확하고 간결하게
2
설명: 무엇을 왜 변경했는지
3
스크린샷: UI 변경 시 첨부
4
체크리스트: 테스트 완료 여부

👀 코드 리뷰 과정

# PR 생성 후 팀원들이 할 수 있는 것들: 1. Files changed 탭에서 코드 변경사항 확인 2. 특정 줄에 댓글 달기 3. 전체적인 피드백 제공 4. 승인 (Approve) 또는 변경 요청 (Request changes)

🔧 PR 관련 자주 묻는 질문

Q: PR 생성 후 추가 작업이 필요하다면?
A: 같은 브랜치에서 계속 작업하고 푸시하면 PR이 자동 업데이트됩니다!
Q: 노란색 PR 알림이 안 뜬다면?
A: Pull requests 탭 → New pull request로 수동 생성하세요.
Q: 이미 Closed된 PR만 보인다면?
A: 새로운 변경사항이니 새 PR을 만들어야 합니다.

🎯 병합 후 정리

# PR 병합 완료 후 로컬 정리 git checkout main git pull origin main git branch -d feature/my-feature # GitHub에서도 브랜치 삭제 (선택사항) # PR 페이지에서 "Delete branch" 버튼 클릭

7
충돌 해결하기

💥 충돌이 발생하는 상황

같은 파일의 같은 부분을 여러 사람이 수정했을 때

예: 철수가 README.md 1번째 줄 수정 → 영희도 README.md 1번째 줄 수정 → 충돌!

🔧 충돌 해결 과정

# 1. 충돌 발생 시 나타나는 메시지 git pull origin main # Auto-merging README.md # CONFLICT (content): Merge conflict in README.md # Automatic merge failed; fix conflicts and then commit the result. # 2. 충돌 상태 확인 git status # You have unmerged paths. # (fix conflicts and run "git commit")

📝 충돌 파일 수정하기

# 충돌 파일을 열면 이런 표시가 나타남: <<<<<<< HEAD 현재 브랜치의 내용 (내가 작업한 내용) ======= 들어오는 변경사항 (다른 사람이 작업한 내용) >>>>>>> branch-name # 해결 방법: # 1. 필요한 내용만 남기고 나머지 삭제 # 2. <<<<<<<, =======, >>>>>>> 표시 모두 삭제 # 3. 최종 결과만 남기기

📋 충돌 해결 단계

1
충돌 파일 열기 (VS Code 추천)
2
충돌 표시 찾기 (<<<<<<< ======= >>>>>>>)
3
필요한 내용 선택/편집
4
충돌 표시 삭제
5
파일 저장

✅ 충돌 해결 완료

# 1. 충돌 해결 후 스테이징 git add . # 2. 병합 커밋 생성 git commit -m "충돌 해결: README.md 병합" # 3. 푸시 git push origin main
충돌 예방 팁:
• 작업 시작 전 항상 git pull
• 자주 커밋하고 푸시하기
• 같은 파일을 동시에 수정하지 않기
• 역할 분담 명확히 하기
VS Code에서 충돌 해결:
충돌 파일을 열면 "Accept Current Change", "Accept Incoming Change", "Accept Both Changes" 버튼이 나타납니다. 클릭하면 쉽게 해결 가능!

# 1. 저장소에 HTML 파일 생성 echo " 우리 팀 프로젝트

안녕하세요! 우리 팀 프로젝트입니다.

GitHub Pages로 배포된 웹사이트입니다.

" > index.html # 2. 커밋 및 푸시 git add index.html git commit -m "웹사이트 추가" git push origin main

🎯 브랜치 명명 규칙

# 기능 개발 feature/login-system feature/user-dashboard feature/payment-integration # 버그 수정 bugfix/header-alignment bugfix/login-validation bugfix/responsive-mobile # 핫픽스 (긴급 수정) hotfix/security-patch hotfix/critical-bug # 실험적 기능 experiment/new-ui-design experiment/ai-integration

💬 커밋 메시지 컨벤션

# 타입: 간결한 설명 feat: 새로운 기능 추가 fix: 버그 수정 docs: 문서 수정 style: 코드 스타일 변경 (포맷팅, 세미콜론 등) refactor: 코드 리팩토링 test: 테스트 추가 chore: 기타 변경사항 # 예시: feat: 사용자 로그인 기능 추가 fix: 헤더 메뉴 클릭 안되는 버그 수정 docs: README.md 설치 가이드 업데이트

🔄 코드 리뷰 가이드라인

리뷰어가 확인할 점:
• 코드 품질 및 가독성
• 버그 가능성
• 성능 이슈

리뷰 받는 사람:
• 피드백 수용 및 수정

🚨 응급 상황 대처법

💥 실수로 main에 직접 푸시했을 때

# 1. 마지막 커밋 취소 (파일 변경사항 유지) git reset --soft HEAD~1 # 2. 새 브랜치 생성 git checkout -b feature/fix-mistake # 3. 다시 커밋 git add . git commit -m "실수 수정: 브랜치로 이동" # 4. PR 생성 git push origin feature/fix-mistake

🔄 잘못된 브랜치에서 작업했을 때

# 1. 현재 작업 임시 저장 git stash # 2. 올바른 브랜치로 이동 git checkout correct-branch # 3. 작업 복원 git stash pop
팀 협업 성공 팁:

• 문제 발생 시 즉시 공유
• 서로의 코드 스타일 존중
• 꾸준한 커뮤니케이션
